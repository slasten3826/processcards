# Strong Pair Table

Статус:

```text
canonical table draft
```

Этот документ — рабочая таблица strong combined readings для minor-пар.

Он не обязан покрывать всю матрицу.
Он обязан быть честным:

- без фальшивого `clear`
- без invented intermediate entities
- без disguised weak-sequences

Читать вместе с:

- [STRONG_COMBINED_LAW.md](./STRONG_COMBINED_LAW.md)

## 1. Read law

Каждая запись ниже описывает:

```text
one combined reading of pair AB
```

а не:

```text
A then B
```

## 2. Status law

В таблице используются только три статуса:

- `clear`
- `rough`
- `unresolved`

Правило использования:

- `clear` = pair-reading уже можно считать executable table-surface
- `rough` = направление понятно, но кодить ещё нельзя
- `unresolved` = честно не знаем pair-reading

## 3. Special family exclusion

`☰ CONNECT` не живёт в этой таблице как ordinary strong pair.

Bridge rule:

```text
whenever a pair contains ☰,
CONNECT_LAW overrides ordinary strong-pair interpretation
```

Для всех пар с `☰` читать:

- [CONNECT_LAW.md](./CONNECT_LAW.md)

Причина:

```text
☰ has its own mechanic
```

Значит `☰ + X` не надо замазывать псевдо-combined payload-ами в этом файле.

## 5. Self-pairs

Self-pairs are outside this table for now.

They are not ordinary symmetric pairs
and need dedicated self-pair law.

Examples:

- `▽▽`
- `☳☳`
- `☰☰`
- `△△`

Operational routing:

- pair-class reading stays outside this file
- duplicate behavior reads through [DUPLICATE_LAW.md](./DUPLICATE_LAW.md)

## 6. Current usage rule

For crystall purposes:

- `clear` pairs may be implemented
- `rough` pairs may guide further table discussion
- `unresolved` pairs must not be faked in code

## 7. Short formula

```text
strong pair table documents only real fused readings,
and stays unresolved where that reading is not yet known
```
