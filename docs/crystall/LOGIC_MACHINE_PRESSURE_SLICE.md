# Logic Machine Pressure Slice

Status:

```text
current design reading
post-LOGIC architecture consequence
```

## Core claim

`LOGIC` is not only another operator payload.

It pressures the machine into admitting
that several zones and interaction surfaces
must become more real than before.

## What LOGIC revealed

`LOGIC` forced the project to treat all of the following
as real machine surfaces, not decorative leftovers:

- `grave`
- revealed public minor matter
- multi-step operator targeting
- public-zone selection outside manifest only

## Grave consequence

Before `LOGIC`,
`grave` could still be treated as a compact discard stack
with weak interaction expectations.

After `LOGIC`,
that is no longer honest enough.

Because `LOGIC` works with revealed public minor matter,
including `grave`,
the game now requires:

```text
grave as a readable selectable machine zone
```

This is why the neutral grave viewer became necessary.

## Operator consequence

`LOGIC` also proves that operator phase
cannot stay a trivial one-click branch forever.

It can require:

1. choose operator
2. choose public target
3. choose hand target
4. resolve swap

So operator phase is already confirmed
as a true structured runtime layer.

## Public-space consequence

`LOGIC` makes a strong distinction visible:

- `hand` is not just another board zone
- public revealed minor space is its own resource
- `grave` belongs to that resource

This helps separate:

- `CHOOSE`
- `LOGIC`
- `CONNECT`
- `CYCLE`

instead of letting them collapse into one generic value engine.

## Why this is healthy

This pressure is not accidental damage.

It is useful.

`LOGIC` forces the project
to formalize what was still vague:

- which zones are actually part of machine interaction
- which of them need readable UI surfaces
- which operators are simple
- which operators require staged target pipelines

## Short formula

```text
LOGIC does not only modify cards.
It forces ProcessCards to admit
that grave and public revealed matter
are real machine surfaces.
```
