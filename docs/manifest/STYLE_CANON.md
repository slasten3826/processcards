## 1. Core Claim

`ProcessCards` не должен выглядеть как:

- generic fantasy card game
- generic sci-fi UI
- generic roguelike deckbuilder

Его визуальная идентичность строится на трёх взаимодействующих системах:

1. operators have fixed canonical colors
2. minor and trump cards use different composition laws
3. the board/UI reads as a machine of zones, not a battlefield of creatures

Short formula:

```text
operators = color language
minors = grammar
trumps = event law
ui = process machine
```

## 3. Color Interaction Law

Operator colors are not random decoration.

Core rule:

- operator hue stays canonical
- pairing changes composition
- composition changes class readability

Short formula:

```text
same colors
different class geometry
different class meaning
```

## 4. Minor Card Composition Law

Minor cards are the grammar layer of the machine.

Current canon:

```text
minor = diagonal operator composition
```

Why diagonal:

- blend
- composition
- transition
- local pair-function

Additional minor rules:

- the two canonical operator colors must stay clear
- diagonal direction may vary physically
  but must not imply different gameplay meaning
- mixed minors should feel technical and operative
- duplicate minors remain visually identical within their class

Minor cards should feel:

- precise
- processual
- technical
- less scenic than trumps
- more like grammar than spectacle

## 5. Trump Card Composition Law

Trump cards are the event-law layer of the machine.

Current canon:

```text
trump = vertical event seam
```

Why vertical:

- dual force
- split world
- law boundary
- confrontation of operator identities

Additional trump rules:

- one strong central event-image
- title large and decisive
- flavor text short, law-like, memorable
- more scenic than minors
- operator colors remain canonical
  but become world-scale forces

Trumps should feel:

- absolute
- iconic
- archetypal
- like named events

## 6. Minor vs Trump Distinction

This difference is structural.

### Minor

- diagonal split
- operator grammar
- local card action
- technical surface

### Trump

- vertical split
- event-law
- state change / ecology shift
- strong central illustration

Short formula:

```text
minor = grammar
trump = law
```

## 7. Typography and Text Canon

`ProcessCards` text should feel:

- machine-like
- restrained
- legible
- ritual-industrial
- not ornate fantasy
- not over-explained mobile UI

### Titles

Card titles should be:

- large
- clean
- high-contrast
- readable at glance

### Flavor text

Flavor text should:

- be short
- sound like a law or archetypal statement
- not read like explanatory rules text
- complete the identity of the card

## 8. UI Style Canon

The finished `ProcessCards` UI should preserve the board architecture of the prototype
while expressing the game as a:

```text
process machine
```

Canonical zones remain:

- `DECK`
- `TARGETS`
- `TRUMP ZONE`
- `MANIFEST / LATENT`
- `RUNTIME`
- `HAND`
- `GRAVE`
- `SYSTEM`
- `LOG`

UI mood should feel like:

- industrial sci-fi console
- machine-ritual panel
- systems terminal
- premium indie strategy product
- dark operational environment

It should not feel like:

- playful mobile card UI
- fantasy tavern table
- generic cyberpunk dashboard clutter

Canonical UI material language:

- deep charcoal / midnight base
- thin luminous borders
- restrained panel glow
- crisp typography
- subtle circuit or machine-line detailing
- elegant slot framing
- dark room / high-contrast screen feeling

Short formula:

```text
ui = machine ritual console
```

## 9. Board-Level Visual Priorities

### `MANIFEST / LATENT`

This is the signature structure of the game.
It must read as:

- one unified machine
- two stacked layers
- five aligned columns
- visible tension between open and hidden state

### `TRUMP ZONE`

Should feel like:

- event residue chamber
- pressure storage
- resolved anomaly archive

### `RUNTIME`

Should feel:

- distinct
- dark
- infrastructural
- special

### `LOG`

Should feel like:

- memory of process
- event trace
- machine readout of history

### `SYSTEM`

Should feel like:

- state monitor
- compact machine status layer
- counters, not lore panel

## 10. Product Style Confirmation

Current screens and validated mockups already confirm:

1. `ProcessCards` is not visually generic
2. it has a recognizable operator-color language
3. the board reads as a process machine, not a creature battlefield
4. the game identity depends on layered visibility
5. finished screens should feel like live machine-state, not decorative menu art

This means the style canon is no longer purely speculative.
It is already visible in accepted mockup direction.
